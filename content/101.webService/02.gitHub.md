# GitHubでdraft Pull requestを作成する

## draft PRの作成方法

通常のPRを作成する手順とほぼ同じ。

1. `Pull Requests` タブからマージ元／マージ先ブランチを選択し、`Create PR` をクリック。
2. PR作成に必要な項目を入力。
3. `Create pull request ▽` をクリックしドロップダウンを表示する。その中から `Create a draft pull request` をクリックすることでdraft PRを作成することができる。

作成後も、`draft` と通常（Ready for review）のステータスを切り替えることができる。

* マージボックスにある `Ready for Review` をクリックすると、レビュー準備完了のステータスにすることができます。
* Reviewers欄にある `Convert to draft` をクリックすると、逆にそのPRのステータスを `draft` にすることができます。

# GitHub ライブラリをGithubに公開する

## GitHubPackagesとは

GitHubPackagesはGitHubのサービスの一つで、まさに今回のようなライブラリや、パッケージなどを共有するためのもの。
mavenやnpm、NuGetなどの形式で公開することができる。
公開範囲も指定でき、一般公開からプライベートやTeam限定など制御することができるとのこと。

今回はJava向けのライブラリを公開するために、Gradleプラグインを使った手順を紹介する。

## GitHubPackagesにライブラリを公開する設定手順

1. **app/build.gradleに設定を追加**する。
2. **Gradleタスクを実行**する。
3. **（任意）GitHub Action**で自動的に更新する。

## app/build.gradleに設定を追加する

`app/build.gradle`に以下を追加します。

```groovy
plugins {
  ...
  id 'maven-publish'
}

publishing {
  ...

  repositories {
    maven {
      name = "GitHubPackages"
      url = "https://maven.pkg.github.com/mkt120/sample_repository"
      credentials {
        username = System.getenv("USER_NAME")
        password = System.getenv("ACCESS_TOKEN")
      }
    }
  }
}

```

* `plugin`に`maven-publish`を追加。ライブラリ公開を簡単に実行してくれるプラグインを組み込む。
* `url`はGitHubのリポジトリURL。
* GitHubのリポジトリに対応したURLを設定。


* `username`、`password`はログイン情報。
* 特にpasswordはログインパスワードだと警告orエラーが発生するので、Githubからアクセストークンを取得しそれを設定する。直書きだとセキュリティ観点でリスクがあるので、Githubの `secret and variables` に設定しておいて、それを参照するようにしておくのが無難。

## Gradleタスクを実行する

前項の設定を追加し、`sync gradle` を実行すると以下のタスクが追加されます。

```bash
publishGprPublicationToGitHubPackagesRepository

```

これをコマンドラインやGradleペインから実行することで、 GithubPackagesへの登録が行われる。

```bash
./gradlew publishGprPublicationToGitHubPackagesRepository

```

（GithubPackageによると）以下のタスクはGithubPackagesへの登録だけでなく、その前段階のライブラリのビルドなども一緒に実行してくれる。

```bash
./gradlew publish

```

## （任意）GitHub Actionで自動的に更新する

ここまでだけでGitHubPackagesにパッケージを登録することができるが、
GitHubにコミットをプッシュしたことを契機にGitHubPackagesに更新を入れるよう、GitHub Actionで自動登録できるようにしておくと効率的に開発できる。

```yaml
#masterブランチにプッシュされたら実行
on:
  push:
    branches: [ "master" ]
  ...

jobs:
  build:
  ...
   
  # name. githubAction のコンソール上で実行内容がわかりやすいように名前付けしておく
  - name: publish library 
    run: ./gradlew publish
    env:
      USER_NAME: ${{ secrets.USER_NAME }}
      ACCESS_TOKEN: ${{ secrets.ACCESS_TOKEN }}

```

# `groupId` 、 `artifactId` 、そして `version` について

Maven / Gradleにおけるプロジェクト識別子。

JavaやKotlin、Android開発などのビルド構成ファイル（`pom.xml` や `build.gradle`）でライブラリを導入したり定義したりする際、
プロジェクトを一意に識別するために以下の3つの要素（座標）が使用される。

## groupId

自身のプロジェクトを、世界の他のすべてのプロジェクトから明確に区別するための「組織・グループの識別ID」。

一般的には、開発元が所有するインターネットのドメイン名を逆順にしたものが使われる。

* 例： `com.example`、 `org.apache`

個人開発であれば、自身のWebサイトのドメインやGitHubのアカウントをベースにしたドメイン（`com.github.ユーザー名` など）を設定するのが無難。

## artifactid

プロジェクトから生成される「成果物（ライブラリやアプリ本体）の名前」。基本的にはプロジェクト名やライブラリ名そのものを指す。
スペースを含めず、小文字のアルファベットとハイフン（`-`）で構成するのが一般的。

* 例： `okhttp`、 `room-runtime`

## version

そのプロジェクトが現在どの開発フェーズにあるかを示す「バージョニング情報」。

プロジェクトの新規作成時には、デフォルトで `1.0-SNAPSHOT` などの値が割り当てられることが多い（`SNAPSHOT` は開発中のバージョンであることを意味する）。表記の厳密なフォーマットは強制されていないが、一般的にはセマンティックバージョニング（`主バージョン.副バージョン.修正バージョン`）に倣って管理される。
